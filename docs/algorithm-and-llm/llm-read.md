# Planner LLM 训练阅读综述与落地方案（对齐系统设计与规划）

## 1. 问题定义

核心问题：
当前系统可通过外部大模型生成 Plan，但在成本、时延、结构稳定性与规则贴合度上存在波动。是否需要训练自有 Planner 模型，以及如何训练，才能与现有 FSM/HITL 契约兼容。

本文给出的结论是：

- 不做从零预训练；
- 做“Planner 专用模型”与“外部强模型兜底”的双路架构；
- 训练目标优先级是可执行性与稳定性，而不是通用问答能力。

## 2. 当前系统基线（截至 2026-03-01）

系统侧已具备：

- Planner `LLM Provider` 可插拔（baseline / OpenAI-compatible）。
- 执行恢复闭环：`retry -> patch -> replan`。
- HITL 审计路径：`WAITING_*` + `PendingAction` + `Decision` + `EventLog`。

系统侧尚待强化：

- Top-K 候选质量与排序稳定性。
- `score_breakdown`、风险/成本门控一致性。
- 失败修复样本的体系化沉淀与评估。

______________________________________________________________________

## 3. 是否自训练：结论与边界

结论：建议训练 7B-14B 规模的 Planner 专用模型（SFT/QLoRA 为主），并保留外部模型回退。

边界：

- 不改变现有 FSM 与 Agent 职责分工。
- 不将“模型判断”替代“结构化校验与状态机约束”。
- 不追求一次性替换全部路径，应按能力分阶段上线。

## 4. 训练目标分解（与系统契约对齐）

| 训练目标     | 对应系统契约                            | 评价指标                     |
| :----------- | :-------------------------------------- | :--------------------------- |
| 结构合法生成 | Plan/PlanPatch/Replan schema            | Schema 合法率                |
| 候选可执行性 | ToolKG 可用性 + I/O 闭包                | 可执行候选率                 |
| 候选可审计性 | `score_breakdown/risk/cost/explanation` | 字段完整率 + 人审通过率      |
| 局部修复能力 | Patch 优先局部最小变更                  | Patch 最小性命中率           |
| 策略调整能力 | Replan 优先保留成功前缀                 | `suffix_replan` 前缀保持率   |
| 在线稳定性   | WAITING\_\* 决策链路 + 回退机制         | 人工介入率、恢复成功率、时延 |

## 5. 数据工程方案（训练样本从系统日志来）

### 5.1 数据来源

- Task 输入与约束
- Planner 候选与最终选择
- StepResult 执行结果
- PendingAction 与 Decision
- EventLog 与 TaskSnapshot

### 5.2 样本单元

每条样本尽量包含：

- `context`：任务目标、约束、已成功前缀
- `candidate_set`：Top-K 候选及打分
- `outcome`：执行结果、失败类型、是否触发 WAITING\_\*
- `decision_trace`：人工选择与理由（若有）

### 5.3 数据清洗与质量门禁

- 剔除 schema 不合法样本。
- 剔除关键字段缺失样本（尤其是 failure_code、risk/cost）。
- 对“成功但不可复现”样本降权。
- 对“失败后修复成功”样本升权（用于 patch/replan 学习）。

## 6. 训练路线（分阶段）

### 阶段 A：SFT（结构与约束优先）

目标：让模型稳定输出合法结构并满足 ToolKG 约束。

建议：

- 先覆盖 Plan 生成，再补 Patch/Replan。
- 以 JSON 结构一致性为主损失目标。

### 阶段 B：偏好优化（DPO/同类方法）

目标：让模型偏好“成功率高、风险低、改动小”的候选。

偏好对构造：

- 正样本：执行成功、风险可接受、修复代价小。
- 负样本：执行失败、违反约束、改动过大。

### 阶段 C：失败修复强化（Reflexion 风格）

目标：把失败反馈转化为高质量 patch/replan 提案。

建议：

- 引入 evaluator，将失败轨迹映射为结构化错误类型与文本反馈。
- 强化“先 patch 后 replan”的行为先验。

### 阶段 D：蒸馏与部署优化

目标：降低推理时延与资源占用，保持稳定性。

建议：

- 蒸馏到更小模型或低比特量化。
- 维持关键字段完整性回归测试。

## 7. 推理与上线架构（双路）

默认路径：

1. 自研 Planner 模型生成候选
1. schema verifier + 约束检查
1. 通过则进入执行/HITL 决策

回退路径：

- 当出现以下条件之一时回退外部强模型：
  - schema 连续不合法
  - 候选可执行率显著下降
  - 连续失败超过阈值
  - 风险等级持续偏高

关键原则：

- 回退只替换“模型调用层”，不改变 FSM/Decision 契约。

## 8. 评估框架与门槛

### 8.1 离线评估

- Schema 合法率 >= 99.5%
- 可执行 Plan 率 >= 95%
- Patch 最小性命中率 >= 80%
- `suffix_replan` 前缀保持率 = 100%

### 8.2 在线评估（对比现有外部模型基线）

- 首轮执行成功率提升（目标 +10%）
- 平均时延下降（目标 >= 30%）
- 人工介入率下降且恢复成功率不下降
- EventLog 可追溯完整率不下降

## 10. 风险与控制

- 风险：训练集分布偏窄，导致上线后泛化不足。
  - 控制：保留外部模型回退，持续采样难例增量训练。
- 风险：模型追求短期成功率，牺牲可解释性。
  - 控制：把 `score_breakdown/explanation` 设为硬性验收字段。
- 风险：训练优化破坏现有执行契约。
  - 控制：上线门禁必须包含 schema/FSM/HITL 回归测试。
